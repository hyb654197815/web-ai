# 前端业务代码定位策略（按需加载）

## 路径约定

- 默认将当前 Skill 根目录下的 `references/` 视为知识文档目录
- 若运行层显式提供 `references_dir`、`knowledge_skill_root` 或等价配置，以其解析结果为准
- `.opencode/skills/` 只是当前 OpenCode 的默认安装布局，不是唯一前缀
- 在不同工具或不同目录架构下迁移时，始终优先使用“当前 Skill 根目录 + references/”这一相对关系，不要把绝对目录写死到提示词或脚本里

## 默认先读入口索引

当用户描述的是“要改哪个业务”而不是“要改哪个文件”时，先读取本 Skill 的 `references/routes.md`，不要一开始就遍历所有 `page-xxx.md`。

`references/routes.md` 提供：
- 业务页面与路由映射
- 页面名称
- 组件文件路径
- 对应页面文档文件名

优先用它回答以下问题：
- 这个业务需求最可能落在哪个页面
- 对应页面源码入口文件是什么
- 是否存在多个候选页面需要继续缩小范围

如果用户已经明确给出路由、页面名称或组件文件路径，可以直接进入对应源码，无需额外加载其他页面文档。

## 业务定位流程

当用户只给出业务描述时，按下面顺序定位：

1. 先从 `routes.md` 匹配页面名称、路由路径、组件文件。
2. 给出 1 到 3 个最可能的页面候选，优先展示：
   - 路由
   - 页面名称
   - 组件文件
   - 对应页面文档
3. 只有在需要进一步确认页面职责、主流程或页面结构时，才读取对应 `references/page-xxx.md`。
4. 确认目标页面后，再进入真实源码文件开展修改或分析。

## 按需读取页面文档

仅当以下情况出现时，再读取对应 `page-xxx.md`：
- 用户只描述业务流程，不确定页面名称
- 需要区分相近页面或相近模块
- 需要结合页面用途、关键操作步骤来判断改动点
- 需要确认页面属于列表页、表单页、详情页、设置页、权限页或报表页

不要一次性读取全部页面文档。

## 代码定位规则

确定目标页面后，优先从 `routes.md` 中的 `组件文件` 进入源码，并按改动类型继续扩展：

- 页面结构、文案、按钮、布局、样式：
  先看页面组件本身，以及同目录下的 `components/`、样式文件、局部子组件。
- 搜索区、表格、筛选、分页、弹窗、抽屉：
  重点看页面组件中引入的表格列定义、表单配置、弹窗组件、列表工具栏组件。
- 表单字段、校验、提交逻辑：
  重点看表单 schema、rules、model、submit handler，以及相关弹窗或详情组件。
- 接口请求、数据加载、保存提交：
  重点看页面组件引入的 `api`、`service`、`request`、`http` 文件。
- 状态管理、缓存、全局联动：
  重点看 `store`、`pinia`、`vuex`、`composables`、`hooks`。
- 权限、菜单、路由守卫、页面跳转：
  重点看路由配置、layout、permission guard、menu 配置。
- 常量、枚举、字典、下拉选项：
  重点看 `constants`、`enums`、`dict`、配置文件。

如果页面组件里已经 import 了依赖文件，优先顺着 import 链继续定位，不要先做全局盲搜。

## 输出要求

帮助前端工程师定位代码时，优先输出：
- 最可能需要修改的入口文件
- 2 到 5 个相关联文件
- 每个文件与业务需求的关系
- 如果存在歧义，说明为什么它们都是候选项

如果 `routes.md` 已经足够定位到页面入口，就先给出入口文件，再决定是否继续展开页面文档或源码细节。

## 文件位置

- 路由索引：当前 Skill 根目录下的 `references/routes.md`
- 页面文档：当前 Skill 根目录下的 `references/page-xxx.md`
- 实际源码入口：以 `references/routes.md` 中的 `组件文件` 列为准
